## 🔧 执行流程（四步法）

### 步骤 1：匹配模板与路径
```
收到任务 → 查阅本表 → 匹配工作流 → 获取：
  ├── 资源来源文件（素材、数据）
  ├── 执行模板文件（输出格式）
  └── 结果归档路径（存储位置）
```

### 步骤 2：按流程执行
- 严格遵循既定步骤，不新增、不简化、不更改
- 遇到问题先查 03-Knowledge，再查外部资源

### 步骤 3：分层沉淀
```
沉淀层级：
├── 主页.md ← 所有工作流精简简介
├── 工作流详情文档 ← 完整流程细则
└── 日志/归档 ← 执行记录
```

### 步骤 4：记忆固化
- 定稿工作流存入 Hermes 长期记忆
- 后续任务优先匹配既定规则

## 📝 WF-05：会话归档细则

### 归档规则

1. **价值预判**：优先用免费模型（如 DeepSeek/GLM）预判会话内容价值，摘要关键内容后再归档
2. **一会话一文件**：每个 Hermes 会话独立一个归档文件，文件名 `YYYY-MM-DD 会话标题.md`
3. **标题一致**：归档文件标题与 Hermes 会话标题一致
4. **时间戳去重**：通过 `last_archived` 字段判断，只归档新内容
5. **排序**：每条归档带 `时间:` 字段，按时间顺序排列
6. **格式**：统一 `问题 → 解决方案 → 解决效果` 三段式
7. **字数**：子会话 ≤500 字；整篇动态上限（500~5000 字）
8. **验证**：归档完成后运行 `verify_archive.py` 验证字数

### 价值预判流程（第一步）

```
会话内容 → 免费模型预判 → 输出：
  ├── 内容价值评级：高/中/低/无
  ├── 关键内容摘要（3-5句话）
  ├── 建议动作：归档 / 跳过 / 仅记录标题
  └── 预估归档字数
```

**预判规则：**
- 高价值（经验/项目/流程/决策）→ 完整归档
- 中价值（有参考性但不紧急）→ 精简归档
- 低价值（重复/常见内容）→ 仅记录标题和日期
- 无价值（问候/调试日志/敏感信息）→ 跳过不归档

**模型选择优先级：**
1. DeepSeek（免费/低价）
2. GLM（免费额度）
3. 本地模型
4. 兜底：直接执行（不预判）

### 文件命名规则

```
98-Archive/
├── 2026-07-04 Obsidian关联确认.md
├── 2026-07-04 Token增加原因与解决方案.md
├── 2026-07-04 简历模板代码推送.md
└── ...
```

### 去重判断（脚本化）

```bash
# 检查指定会话是否有新内容
python 04-Resource/scripts/check_archive_new.py "Obsidian关联确认" 2026-07-04T16:30

# 列出所有已归档会话
python 04-Resource/scripts/check_archive_new.py --list

# 输出：
# 📋 已归档会话 (3 个):
#   2026-07-04 Obsidian关联确认 (最后归档: 2026-07-04T15:30)
#   2026-07-04 Token增加原因与解决方案 (最后归档: 2026-07-04T14:00)
```

### 字数决策流程

```
原始内容 <2000字  → 目标 500~1000 字（内容稀疏，精炼核心）
原始内容 2000~8000字 → 目标 1000~3000 字（内容适中，保留主线）
原始内容 >8000字  → 目标 3000~5000 字（内容丰富，充分展开）
```

### 执行流程

```
1. 价值预判：用免费模型扫描会话，输出价值评级+关键摘要
2. 仅处理高/中价值会话，低/无价值跳过
3. 运行 check_archive_new.py --list 查看已归档会话
4. 逐个 check_archive_new.py <标题> <时间> 判断是否有新内容
5. 对需要更新的会话，基于预判摘要精炼为「问题→方案→效果」
6. 单块 ≤500 字，追加到对应归档文件，更新 last_archived
7. 运行 verify_archive.py 验证字数
8. 不通过 → 压缩后重试
```

### 有效内容筛选

归档时只保留以下类型内容，过滤无意义对话：

| 保留 | 过滤 |
|------|------|
| 经验总结、踩坑记录 | 重复确认、简单问候 |
| 项目进展、技术方案 | 工具输出、调试日志 |
| 流程规范、最佳实践 | 纯展示、无结论讨论 |
| 个人信息、偏好设置 | 密码、token、敏感信息 |
| 决策依据、权衡分析 | 试错过程、无效尝试 |

### 筛选原则

```
原文 → 判断是否有持久价值 → 有价值则保留，否则丢弃
                            ↓
                     有价值 = 未来某天回看仍需要
```

### 会话归档模板


### 📄 [github_forbid] 文件命名规则

对于**不需要归档**（archive: ❌）的任务，输出文件命名遵循以下规则：

```
[github_forbid]任务描述.md
```

**规则：**
1. 文件名以 `[github_forbid]` 开头
2. `.gitignore` 中已配置 `[github_forbid]*` 自动忽略
3. git add/commit/push 时自动跳过，无需手动排除
4. 文件仍保留在本地，可手动查看

**示例：**
```markdown
# 编码任务输出（临时代码，不归档不推送）
文件名: [github_forbid]数据清洗脚本调试.md
4维度: WF-11 | 按需 | ❌不归档 | ❌不推送
```
