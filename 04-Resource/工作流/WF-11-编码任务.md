# WF-11 编码任务

## 🎯 工作流目的

规范化编码任务的执行流程，从需求分析到代码交付，确保代码质量、可维护性和文档完整性。支持从简单脚本到复杂功能模块的各类编码需求。

## ⚡ 触发条件

- 用户提出明确的编码需求（新功能、Bug 修复、重构等）
- 技术调研完成后需要进行 POC/Demo 验证时
- 需要编写自动化脚本或工具时
- 用户提出"写一个XX"、"实现XX功能"、"修复XX问题"等需求时

## 📋 执行步骤

### 第一步：需求分析与确认
1. 理解编码任务的目标和约束条件
2. 确认技术栈、运行环境和依赖要求
3. 明确输入输出规格和边界条件
4. 评估任务复杂度，决定执行策略：
   - 简单任务（<100 行）：直接编码
   - 中等任务（100-500 行）：先规划后编码
   - 复杂任务（>500 行）：创建实施计划，分步执行
5. 如需要，创建任务计划文档

### 第二步：架构设计（中等及以上任务）
1. 设计模块结构和接口定义
2. 确定数据流和控制流
3. 选择合适的设计模式和最佳实践
4. 识别潜在的技术风险和解决方案
5. 与用户确认设计方案

### 第三步：编码实现
1. 创建项目结构和配置文件
2. 按模块逐步实现核心功能
3. 遵循编码规范：
   - 清晰的命名约定
   - 适当的注释和文档字符串
   - 错误处理和边界检查
   - 遵循语言/框架最佳实践
4. 每个模块完成后进行基本验证

### 第四步：测试与验证
1. 编写单元测试（关键逻辑必覆盖）
2. 进行集成测试和端到端验证
3. 边界条件和异常路径测试
4. 性能测试（如有性能要求）
5. 确保所有测试通过

### 第五步：文档与交付
1. 编写/更新 README 文档
2. 添加使用示例和 API 文档
3. 记录已知限制和注意事项
4. 生成变更日志（CHANGELOG）
5. 如需要，执行 Git 提交（遵循 commit 规范）
6. 向用户交付成果，说明使用方法

## 📝 备注

- 遵循 test-driven-development 技能的 RED-GREEN-REFACTOR 原则
- 对于涉及系统修改的编码任务，先创建备份
- 敏感信息（密码、密钥等）禁止硬编码，使用环境变量
- 复杂任务应委托给专业编码 Agent（如 Claude Code、Codex）
- 编码过程中遇到阻塞问题，及时与用户沟通
- 代码审查可参考 requesting-code-review 技能
- 所有代码应有基本的错误处理，不允许静默失败
