# WF-12 模型编排执行细则（v3.0）

## 🎯 工作流目的

通过三层编排架构，实现不同模型在不同任务上的最优分工，发挥各模型特长，降低成本，提升效率。

---

## ⚡ 触发条件

- 每次任务执行时（模型编排是每次任务的必经步骤）
- 任务需要多个模型协作时
- 需要轻量模型做预处理、大模型做深度分析时

---

## 🏗 三层编排架构

```
┌─────────────────────────────────────────────────────┐
│  第一层：主模型规划（deepseek）                       │
│  读取任务 → 分析复杂度 → 决定编排策略                 │
└───────────────────────┬─────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────┐
│  第二层：编排选择                                     │
│                                                     │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────┐ │
│  │ MoA 模式     │  │ delegate 模式 │  │ 混合模式  │ │
│  │ 多模型聚合   │  │ 同模型并行   │  │ MoA+委派  │ │
│  └──────┬───────┘  └──────┬───────┘  └─────┬─────┘ │
└─────────┼─────────────────┼────────────────┼───────┘
          │                 │                │
          ▼                 ▼                ▼
┌─────────────────────────────────────────────────────┐
│  第三层：执行 + 聚合                                  │
│  收集各模型/子代理输出 → 聚合 → 归档                  │
└─────────────────────────────────────────────────────┘
```

---

## 📋 第一层：主模型规划

主模型收到任务后，执行以下分析：

### 1.1 任务分析

```
任务到来
├── 需要多视角分析？（对比、评估、决策）
│   └── 是 → 选 MoA 模式
├── 需要并行处理？（多个独立子任务）
│   └── 是 → 选 delegate 模式
├── 需要轻量预处理 + 深度分析？
│   └── 是 → 选混合模式（MoA + delegate）
└── 简单任务？
    └── 是 → 主模型直接执行，不编排
```

### 1.2 模型能力画像

| 模型 | 擅长 | 不擅长 | 成本 |
|:----:|:-----|:-------|:----:|
| **GLM** | 文本总结、信息提取、格式转换、轻量压缩 | 复杂推理、长文分析 | 免费 |
| **DeepSeek** | 编码、技术分析、中等推理、代码审查 | 创意写作 | 低 |
| **Mimo** | 通用对话、复杂推理、长文理解、架构设计 | — | 中 |

---

## 📋 第二层：编排模式选择

### 模式 A：MoA（多模型聚合）

**适用场景**：需要不同模型从不同角度分析同一问题

**原理**：
```
同一个 prompt
    │
    ├──→ GLM（轻量视角：快速总结、关键词提取）
    ├──→ DeepSeek（技术视角：深度分析、逻辑推理）
    │         （并行执行）
    ▼
Mimo（聚合：整合所有视角，输出最终结果）
```

**MoA Preset 配置**：

```yaml
# Preset 1: 轻量聚合（GLM 做参考，DeepSeek 聚合）
lightweight:
  reference_models:
    - provider: glm
      model: glm-4-flash
  aggregator:
    provider: code
    model: deepseek-v4-flash
  use_case: 文本压缩、信息提取、快速总结

# Preset 2: 标准聚合（GLM + DeepSeek 做参考，Mimo 聚合）
standard:
  reference_models:
    - provider: glm
      model: glm-4-flash
    - provider: code
      model: deepseek-v4-flash
  aggregator:
    provider: xiaomi
    model: mimo-v2.5-pro
  use_case: 技术调研、方案对比、多维度分析

# Preset 3: 深度聚合（DeepSeek + Mimo 做参考，Mimo 聚合）
deep:
  reference_models:
    - provider: code
      model: deepseek-v4-flash
    - provider: xiaomi
      model: mimo-v2.5-pro
  aggregator:
    provider: xiaomi
    model: mimo-v2.5-pro
  use_case: 复杂推理、架构设计、长文档分析
```

### 模式 B：delegate_task（同模型并行）

**适用场景**：需要把一个大任务拆成多个独立子任务并行处理

**原理**：
```
大任务
    │
    ├──→ 子代理1（处理子任务A）
    ├──→ 子代理2（处理子任务B）
    ├──→ 子代理3（处理子任务C）
    │         （并行执行，同模型）
    ▼
主代理汇总结果
```

**使用场景**：
- 批量文件处理
- 多个独立调研任务并行
- 代码审查（多个文件同时审查）

### 模式 C：混合模式（MoA + delegate）

**适用场景**：需要轻量模型预处理 + 大模型深度分析

**原理**：
```
长文档/大项目
    │
    ▼
第一步（delegate 并行）：
    ├──→ GLM 子代理1：提取第1-3章摘要
    ├──→ GLM 子代理2：提取第4-6章摘要
    ├──→ GLM 子代理3：提取第7-9章摘要
    │         （轻量模型并行压缩）
    ▼
第二步（MoA 聚合）：
    所有摘要 → DeepSeek 深度分析 → Mimo 聚合输出
```

**典型场景**：
- 长文档压缩（GLM 分段压缩 → DeepSeek 深度分析）
- 大项目代码审查（GLM 提取关键代码 → DeepSeek 审查）
- 多源信息整合（GLM 提取各源要点 → Mimo 综合判断）

---

## 📋 第三层：执行与聚合

### 执行流程

```
1. 主模型确定编排模式（MoA / delegate / 混合）
2. 根据模式执行：
   - MoA：调用 /moa <prompt> 或配置 MoA provider
   - delegate：调用 delegate_task({tasks: [...]})
   - 混合：先 delegate（轻量预处理），再 MoA（深度聚合）
3. 收集各模型/子代理输出
4. 聚合结果：
   - MoA：聚合模型自动完成
   - delegate：主代理手动汇总
   - 混合：delegate 输出作为 MoA 输入
5. 归档输出
```

### 聚合策略

| 输入类型 | 聚合方式 | 输出 |
|:---------|:---------|:-----|
| 多个视角分析 | 取交集 + 补充 | 综合结论 |
| 多个子任务结果 | 拼接 + 去重 | 完整报告 |
| 多段压缩摘要 | 串联 + 逻辑梳理 | 压缩文档 |

```

│   ├── 需要多视角？ → 选 MoA
│   ├── 需要并行处理？ → 选 delegate
│   └── 需要预处理+深度分析？ → 选混合模式

│   │
│   ├── MoA 模式

│   │   └── 深度聚合（DeepSeek+Mimo→Mimo）
│   │
│   ├── delegate 模式
```

## 📝 备注

- MoA 实现真正的多模型分工（GLM 做轻量，DeepSeek 做深度）
- delegate_task 用于同模型并行（不能指定不同模型）
- 混合模式是复杂任务的最优选择
- 轻量模型（GLM）适合：总结、提取、压缩、格式化
- 大模型（DeepSeek/Mimo）适合：推理、分析、架构、代码
